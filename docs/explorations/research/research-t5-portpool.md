## TL;DR Decision

**Keep the v1 design as specified in §4.2 / §5.3: a static pre-allocated pool of 80 WireGuard instances (ports 51820–51899) in a single `mitmdump` process — but make two adjustments:**

1. **Reduce the default pool to a configurable size (recommend default 32, hard cap 80).** 80 idle instances cost only ~8 MiB and <1 s over a 1-instance baseline, so the *resource* argument for shrinking is weak — but a smaller default narrows the exposed-UDP-port surface and matches realistic desktop concurrency (the spec already caps VM memory at "50% of host RAM, max 32 GiB", so a laptop cannot run anywhere near 80 Podman Machine microVMs). The pool size is already listed as a `config.yaml` knob ("port pool range", §9). Make 51820+`pool_size` the range; keep 80 as the ceiling.
2. **Do NOT adopt dynamic runtime add/remove for v1** even though the spike proves it works flawlessly (see §2). Static pre-allocation is simpler, has no measured downside at this scale, and keeps the client-config-capture flow (parse the fenced blocks once at daemon start) trivial. Record dynamic add/remove as the *validated* v2 path (it de-risks Risk #3's fallback).

The empirical numbers below **retire Open Risk #3** ("startup latency or memory overhead ... needs benchmarking"): overhead is negligible.

## 1. Startup latency + memory of N idle WG instances (empirical)

One `mitmdump` process, N repeated `--mode wireguard:<path>@<port>` flags, keys auto-generated, **no connected clients**. "Ready" = all N `"WireGuard server listening at *:<port>"` log lines emitted (each instance also emits its client config fenced by a line of exactly 60 hyphens — confirmed `endpoints_emitted == N`). Peak RSS sampled via `ps -o rss`.

| Instances | Time-to-ready | Peak RSS | Marginal RSS/instance |
|----------:|--------------:|---------:|----------------------:|
| 1         | 0.63 s        | 92.8 MiB | (baseline)            |
| 10        | 0.50 s        | 94.3 MiB | ~0.17 MiB             |
| 40        | 0.63 s        | 97.2 MiB | ~0.11 MiB             |
| 80        | 0.82 s        | 101.9 MiB| ~0.11 MiB             |

Cross-checked with a manual 80-instance run: RSS **104.3 MiB flat** from t=1 s through t=12 s (no growth while idle), all **80/80** listening, **80** distinct `Endpoint` ports 51820–51899 emitted, **80** closing fences. Empty `--mode regular` baseline mitmdump ≈ 91.6 MiB.

**Findings:**
- **Memory:** the ~92 MiB is the Python+mitmproxy interpreter/runtime floor. WG instances add ~**0.1 MiB each** → **80 idle instances cost only ~8–13 MiB total** over baseline. Negligible for a desktop tool.
- **Startup latency:** sub-second for 80 instances (0.82 s wall, of which most is interpreter start). Adding 79 instances over 1 added <0.2 s. Not user-perceptible; the daemon starts once and stays resident.
- **Idle overhead:** zero growth over 12 s idle — confirms the spec's "no clients = no traffic = idle sockets" assumption.

> These numbers are Python-side RSS on macOS. The Rust `mitmproxy_rs` WG servers run in the same process; RSS captures them. Absolute floor will differ slightly on Linux/Windows but the *marginal* per-instance cost is the load-bearing number and it is tiny.

## 2. Runtime add/remove without process restart — **SUPPORTED** (source + live spike)

**Yes, mitmproxy supports adding and removing WG server instances at runtime without restarting the process, via the options API.** This is not a hack — it is the exact mechanism `configure()` already uses.

**Source evidence (mitmproxy 12.2.3):**
- `mitmproxy/addons/proxyserver.py:258-304` — `Proxyserver.configure()`: when the `"mode"` option changes and the server `is_running`, it schedules `self.servers.update(modes)`.
- `mitmproxy/addons/proxyserver.py:55-99` — `Servers.update(modes)`: diffs the new mode list against `self._instances`; **starts** instances for newly-added specs (`ServerInstance.make(...).start()`), **stops** instances whose spec is no longer present (`s.stop()`), reuses unchanged ones. Stops run *before* starts "to free ports before starting new servers."
- `mitmproxy/proxy/mode_servers.py:346-379` — each WG instance's `_start()` auto-creates its keys file if absent and logs the client config; `stop()` tears down the UDP socket.
- The `mode` option is `Sequence[str]`, "Can be passed multiple times" (`options.py:108-124`).

**Live spike (addon-driven, one process, no restart):**
```
STEP0        modes=[...@51820]                servers=1   (WireGuard server listening at *:51820)
STEP1(add)   modes=[...@51820, ...@51821]     servers=2   (WireGuard server listening at *:51821)
STEP2(remove) modes=[...@51821]               servers=1   (WireGuard server at *:51820 stopped.)
```
An addon called `ctx.options.update(mode=[...])` at runtime; server count grew 1→2 (with @51821 emitting a fresh client config) then shrank 2→1 (with @51820 cleanly "stopped"). **No process restart, no dropped state on the surviving instance.**

**Static vs dynamic trade-off:**

| | Static pre-alloc (80) | Dynamic add/remove |
|---|---|---|
| Complexity | Lowest — build flags once, capture configs once at boot | Must drive `options.update` from Go (over the addon/UDS boundary or a control channel), handle per-add config capture, re-parse fences per add |
| Config capture | One-shot parse of N fenced blocks at daemon start (§5.3) | Must capture each new instance's config on demand |
| Cost at 80 | ~8–13 MiB, <1 s (measured) | Grows/shrinks with live sandboxes |
| UDP surface | 80 bound ports always | only-in-use ports bound |
| Failure modes | none observed | update is async (`create_task`), needs readiness synchronization before `ppp run` hands the client config to the VM |

**Recommendation:** static for v1 (the measured cost removes the only reason to prefer dynamic); dynamic is now a *validated* v2 option and a proven fallback for Risk #3, so §11 "Future — dynamic mitmdump server add/remove" can be upgraded from "could" to "confirmed feasible via the options API."

## 3. Port + inner-IP allocation, reliable freeing, crash cleanup

### Allocation (current spec, §4.2 / §5.3 / §6.1b)
- Ports 51820..51820+pool-1; inner IP `10.0.0.<N>` where **N = port − 51819** (so 51820→10.0.0.1). Registry: `$PPP_DATA/port-registry.json`, guarded by `$PPP_DATA/state.lock` (flock).
- `ppp run`: acquire lock → pick lowest free port → mark allocated → rewrite that port's client config (`Address = 10.0.0.<N>/32`, `Endpoint = <host>:<port>`) → release lock.
- **Guardrail confirmed by spike:** mitmdump **refuses duplicate ports** — starting two `--mode wireguard@51820` exits immediately with `Cannot spawn multiple servers on the same address: *:51820` (`proxyserver.py:282-287`). So a double-allocation bug fails loud at daemon start rather than silently colliding. Good backstop; do not rely on it for correctness — the registry + flock is the real allocator.

### Freeing on `ppp rm` (§6.5)
`ppp rm` already frees the port in the registry. **The gap is the crash path** (Risk #4: "`ppp rm` must reliably free ports even on crash"). Recommended approach — make the registry self-healing so a stale entry is never fatal:

1. **Registry entry = a claim, not ground truth.** Each `port-registry.json` entry SHOULD store `{port, inner_ip, sandbox_name, machine_name, allocated_at, pid_of_run?}`. The *authoritative* liveness signal is **`podman machine list`** (does a machine named `<sandbox_name>` still exist?) plus presence of `$PPP_DATA/sandboxes/<name>/`.
2. **Reconcile on daemon start and on every allocate** (under `state.lock`): for each "allocated" entry, if its Podman Machine is gone **and** its sandbox dir is gone (or marked removed), reclaim the port. This is the crash-recovery hook analogous to §10.3 — extend "mitmdump dies" recovery to also reconcile the port registry.
3. **`ppp rm` ordering (make free idempotent & crash-safe):** write a `removing` tombstone to the registry entry *before* `podman machine rm`; after a successful rm (or if the machine is already absent) delete the entry. If the process dies mid-rm, the reconciler on next start sees `removing` + no machine → completes the free. `ppp rm -f` must free the port even when `podman machine rm` fails/hangs (spec already says "even remove if machine is unresponsive").
4. **`ppp diagnose`** already surfaces "port registry state" (§10.4) — add a `ppp diagnose --reconcile` (or fold into daemon start) that reports/repairs stale entries. This directly closes Risk #4.
5. **Inner-IP freeing is coupled to the port** (deterministic N ↔ port map), so freeing the port frees the IP automatically — no separate IP registry, no independent exhaustion (retires the second half of Risk #4: "80 ports × 80 IPs"; there is really only one pool of 80).

### Key material on free/reuse
When a port is freed and later re-allocated to a *different* sandbox, its `$PPP_DATA/wg/keys-<port>.conf` keypair is **reused**. That is acceptable because sandbox identity is the listen port bound to that keypair (§3.1), and the two sandboxes never coexist on the port. **Do not** regenerate keys on every rm (it would force re-capturing that instance's client config). If a security policy later wants per-lifetime key rotation, that is the dynamic-add/remove path (§2), where a fresh `_start()` generates a fresh config — another point in dynamic's favor for v2.

## Residual uncertainty

1. **Cross-platform RSS floor.** Numbers are macOS/Python 3.14. Linux (the primary WSL/qemu targets) and Windows floors will differ; re-measure the *marginal* per-instance cost on Linux before ship. Expectation: still ~0.1 MiB/instance (it is a UDP socket + small Rust state).
2. **`process_outgoing_packet` "fall back to first peer" under 80-instance load** (Risk #10 residual, `wireguard.rs:326-333`) — not exercised here (idle, no clients). Must be load-tested with multiple *connected* guests before trusting return-path behavior; unrelated to pool sizing but relevant to running 80 in one process.
3. **Dynamic-add readiness synchronization** (if v2): `options.update` schedules an async task; Go must wait for the new instance's client config to be logged before handing `wg0.conf` to the VM. The spike used fixed sleeps; a real impl needs a readiness signal (poll `len(proxyserver.servers)` / watch for the fence).
4. **Realistic max concurrency** was assumed, not measured — how many Podman Machine microVMs a typical dev host sustains. This bounds the *useful* pool size and justifies a default (32) below the 80 ceiling; validate against §5.1 memory math on target hardware.
5. **Config-format fragility** (Risk #8) is orthogonal but interacts: static capture parses all N fenced blocks once at boot; a format change breaks all at once (loud). Pin mitmproxy 12.2.3 as already specified.
